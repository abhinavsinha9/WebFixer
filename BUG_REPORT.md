# How to Read a Bug Report

This document explains the structure and usage of the Bug Reports generated by BugFinder.

## Report Structure

### 1. Issue Identifier
- **Bug ID:** A unique string (e.g., `BUG-12345`) for tracking.
- **Title:** A concise summary of the issue.

### 2. Classification
- **Severity:** `Critical`, `High`, `Medium`, `Low`, or `Info`. 
  - *Critical* issues break core functionality or pose severe security risks.
  - *Info* issues are suggestions or code smells.
- **Category:** The domain of the issue (`Performance`, `Accessibility`, `Security`, `SEO`, `Code-Quality`).

### 3. Details
- **Description:** A detailed explanation of why the issue was flagged.
- **Affected File & Line:** Exactly where the issue was found (e.g., `src/components/Button.jsx:45`).
- **Code Snippet:** The surrounding code block for context.

### 4. Resolution
- **Root Cause:** The underlying reason for the issue.
- **Suggested Fix:** AI-generated or rule-based guidance on how to fix the code.

## Export Formats

BugFinder allows you to export bug reports in three formats:
1. **PDF:** Best for sharing with stakeholders and high-level reviews. Includes charts and visual scoring.
2. **CSV:** Best for importing into project management tools (Jira, Linear).
3. **Excel:** Best for detailed sorting and manual triage by QA teams.
